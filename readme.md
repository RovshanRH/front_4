# Отчет по выполнению практик 7-11: Магазин Электроники

## Введение

Учебный проект по дисциплине «Фронтенд и бэкенд разработка», выполненный в рамках практических занятий 7, 8, 9, 10 и 11.

Проект реализует полноценный клиент-серверный сценарий:

- регистрация и вход пользователей;
- выдача и обновление JWT-токенов;
- разграничение доступа по ролям (RBAC);
- управление товарами;
- управление пользователями (для администратора).

Фронтенд построен на React (Vite), бэкенд - на Node.js + Express.
Данные хранятся в памяти сервера (массивы пользователей, товаров и refresh-токенов), поэтому после перезапуска сервера изменения сбрасываются.

## Структура проекта

- `client/` - клиентская часть (React + Vite + SCSS).
- `server/` - API-сервер (Express, CORS, Swagger, JWT, bcrypt).
- `Практики/` - методические материалы по практическим занятиям 7-11.
- корневой `package.json` - общий запуск клиента и сервера одной командой.

## Инструкция по запуску

### 1) Установка зависимостей

В корне проекта:

```bash
npm install
```

В папке клиента:

```bash
npm --prefix client install
```

В папке сервера:

```bash
npm --prefix server install
```

### 2) Запуск в режиме разработки

Из корня проекта:

```bash
npm run dev
```

Эта команда одновременно запускает:

- сервер на `http://localhost:4000`;
- клиент (Vite) на `http://localhost:5173`.

- Клиент: `http://localhost:5173`
- API: `http://localhost:4000/api`
- Swagger UI: `http://localhost:4000/api-docs`
- OpenAPI JSON: `http://localhost:4000/swagger.json`

### 4) Демо-учетные записи

- Администратор: `admin@example.com` / `admin123`
- Продавец: `seller@example.com` / `seller123`

### Практика 7. Базовая аутентификация и хеширование паролей

Требования практики:

- регистрация пользователя;
- вход в систему;
- безопасное хранение пароля через хеширование.

Реализация в проекте:

- добавлен маршрут `POST /api/auth/register`;
- добавлен маршрут `POST /api/auth/login`;
- пароли пользователей хешируются через `bcrypt` перед сохранением;
- при входе используется сравнение пароля с хешем;
- данные пользователя не возвращают пароль/хеш наружу (публичная модель пользователя).

### Практика 8. JWT и защищенные маршруты

Требования практики:

- выдача JWT после успешного входа;
- проверка токена в middleware;
- доступ к защищенным маршрутам только для аутентифицированных пользователей.

Реализация в проекте:

- при `POST /api/auth/login` возвращается `accessToken`;
- реализован `authMiddleware`, который проверяет заголовок `Authorization: Bearer <token>`;
- создан защищенный маршрут `GET /api/auth/me`;
- токен содержит `sub`, `email`, `role` и срок жизни.

### Практика 9. Refresh-токены

Требования практики:

- выдача пары `accessToken + refreshToken`;
- добавление маршрута обновления токенов;
- ротация refresh-токена.

Реализация в проекте:

- `POST /api/auth/login` возвращает оба токена;
- добавлен `POST /api/auth/refresh`;
- refresh-токены хранятся в памяти сервера (`Set`);
- при обновлении старый refresh удаляется и выпускается новая пара токенов;
- refresh-токен принимается из заголовка `x-refresh-token` (и поддерживается fallback через `Authorization: Bearer`).

### Практика 10. Фронтенд, хранение токенов и автообновление сессии

Требования практики:

- реализовать фронтенд для регистрации, входа и работы с товарами;
- хранить токены на клиенте;
- автоматически обновлять токены при 401.

Реализация в проекте:

- фронтенд на React содержит формы регистрации и входа;
- токены сохраняются в `localStorage`;
- в `axios` настроены interceptors:
  - request interceptor автоматически подставляет `Authorization`;
  - response interceptor на 401 делает запрос к `/api/auth/refresh`, сохраняет новую пару и повторяет исходный запрос;
- реализованы операции с товарами: просмотр списка, просмотр по id, создание, редактирование, удаление (в зависимости от роли);
- подключен Swagger для удобной проверки API.

### Практика 11. RBAC (разграничение прав по ролям)

Требования практики:

- ввести роли (пользователь, продавец, администратор);
- ограничить доступ к маршрутам по ролям;
- реализовать управление пользователями для администратора.

Реализация в проекте:

- реализованы роли: `user`, `seller`, `admin`;
- создан `roleMiddleware(allowedRoles)`;
- доступ к маршрутам настроен по таблице прав:
  - `GET /api/auth/me` - user/seller/admin;
  - `GET /api/users`, `GET/PUT/DELETE /api/users/:id` - только admin;
  - `POST /api/products`, `PUT /api/products/:id` - seller/admin;
  - `GET /api/products`, `GET /api/products/:id` - user/seller/admin;
  - `DELETE /api/products/:id` - только admin;
- блокировка пользователя (`DELETE /api/users/:id`) выставляет `is_blocked = true` и инвалидирует его refresh-токены;
- фронтенд скрывает/показывает функции управления в зависимости от роли (товары и пользователи).

## Проверка функциональности

Рекомендуемый сценарий проверки:

1. Зарегистрировать обычного пользователя и убедиться, что ему доступен только просмотр товаров.
2. Войти под продавцом и проверить создание/редактирование товаров.
3. Войти под администратором и проверить удаление товаров и управление пользователями.
4. Проверить автообновление access-токена при истечении сессии.
